Keeping Things In Sync (Or Why We Can’t Have Good Things)

Nikhil Pallamreddy

2026-05-05

When you have two related pieces of code that need to stay consistent—say, a data structure and its serialization logic, or an API and its documentation—you face a fundamental problem: how do you keep them from drifting apart?

The Cowboy Approach

Just… don't worry about it. Write your struct, write your JSON handler, ship it. If they drift apart, you'll find out in production when a customer complains. This works great until it doesn't, which is usually around 2 AM on a Friday.

The appeal here is velocity. No ceremony, no tooling, no build step that takes three minutes to regenerate files. The downside is that you're essentially gambling that you (or worse, your coworker) will remember to update both places. Spoiler alert: you won't. Someone once widened an int32 to int64 in a C++ parser for Chromium's session history format and missed the corresponding write path. No crash, no assertion—just silently corrupted data downstream that I found by coincidence.

Document It

Write it down! Create a spec, maintain a README, leave a comment that says "IMPORTANT: if you change X, also change Y." This is what professionals do, and it works exactly as well as you'd expect from a system that relies on people reading documentation.

The documentation approach acknowledges that keeping things in sync is important but solves it with process instead of automation. It's better than nothing, which is damning with faint praise. At least when things break, you can point to the document and say "well, it says right here…" We documented the int32/int64 invariant from the cowboy era. An LLM-generated patch missed it, same silent corruption, same broken downstream data.

AST Analysis and Linting

Now we're getting serious. Use static analysis tools to parse your code's abstract syntax tree and verify invariants. Custom eslint rules can enforce that every enum variant has a corresponding case in your handler. Clippy can yell at you when your Rust implementations diverge from expectations.

This catches mistakes before they reach production, which is a meaningful upgrade from the cowboy approach. The limitation is that AST analysis can only check relationships it's been explicitly taught about. You need to write (or configure) the rules, and the tools are only as smart as their rule sets.

Code Tests

Testing is the boring, reliable workhorse of the sync problem. Write tests that verify the relationship between your related pieces of code. If your API handler doesn't match your OpenAPI spec, the test fails. If your database schema doesn't match your ORM models, the test fails.

Tests are great because they're explicit about what should be true, they run automatically, and they prevent regressions. A round-trip test on the session history parser finally killed the int32/int64 bug class for good—we wrote it once and let it be. The downside is that you're maintaining three things now: the original code, the related code, and the tests that verify they match. Also, tests only verify properties you thought to test, which means the creative ways things can desync will surprise you.

Runtime Validation and Contract Testing

Instead of writing separate assertions about shape, define the shape once and make it the bouncer at every door. Zod parses unknown data against a schema at the edge of your system—if the shape is wrong, it fails immediately with a clear error, not three layers deep in business logic. Pact takes this further across services: consumer and provider independently verify against a shared contract, so neither side can drift without the other knowing.

The schema is both the documentation and the enforcement—you can't forget to update the check because the check is the definition. Unlike code generation, you're still writing both sides; you're just guaranteeing they agree on the boundary. The tradeoff is runtime cost and the fact that you only catch mismatches when the code actually executes—a deployed typo in a rarely-hit path can still hide until a user wanders into it six months later.

Derive and Code Generation

Why maintain two things when you can maintain one thing and generate the other? Define your protocol buffer schema and generate code for six languages. Write your database schema and generate your ORM models. Maintain your OpenAPI spec and generate your client libraries.

Code generation takes one source of truth and mechanically produces everything else. When the source changes, you regenerate, and consistency is guaranteed by construction. Go uses this pattern extensively (go generate, protoc-gen-go, sqlc). The tradeoff is that you need to commit to the generator's opinions about what the generated code looks like, and debugging generated code is about as fun as it sounds.

The classic problem: the generated code is either in version control (and your diffs are polluted with mechanical changes) or it's not (and new developers can't build the project without installing seven tools).

Schema Registries

Code generation's bigger sibling: the source of truth doesn't live in any single codebase—it lives in a registry that multiple services validate against. Kafka's Schema Registry enforces that producers and consumers agree on Avro/Protobuf schemas before messages ever hit the wire. API gateways can reject requests that don't match a registered OpenAPI spec. It's like a treaty with actual enforcement mechanisms instead of strongly worded letters.

No single team can unilaterally break the contract. Schema evolution rules (backward/forward compatibility) are enforced centrally, and violations are caught at deploy or publish time, not in production. The tradeoff is infrastructure: you now depend on an external system for correctness, versioning gets political across teams, and local development requires either a running registry or a convincing fake. Also, good luck getting three teams to agree on a schema evolution strategy—you've traded code drift for committee drift.

Auto-Derive and Type-Level Programming

If schema registries solve the problem with infrastructure, this solves it with math. The final boss. Instead of generating code as a separate build step, use the language's type system or metaprogramming facilities to derive implementations automatically. Rust's derive macros mean you add a field to your struct and serialization, comparison, hashing, and debug formatting all update automatically—you literally cannot forget. Haskell's generics let you write functions that work for any type with the right shape. Scala's implicit derivation summons implementations from the type system itself.

This is "code as data" taken seriously. Your types carry enough information that the compiler can figure out how to serialize them, compare them, or hash them without you writing anything. It's like code generation but at compile time, and the generated code is checked by the same type system that generated it.

The downside is that you're now programming in two languages: the surface language and the type-level meta-language. When something goes wrong, the error messages are arcane because the compiler is trying to explain why it couldn't derive a thing you never wrote in the first place. Your build times bloat as the compiler does more work at compile time—Rust projects heavy on derive macros are notorious for this. The tooling struggles too: autocomplete and go-to-definition get confused when half the code doesn't exist until compilation. And this only works in languages with sufficiently fancy type systems, which rules out a lot of mainstream options. This is why we can't have good things—the final boss demands you pay in language complexity, build minutes, and tooling gaps.

Choosing Your Fighter

As you move down the spectrum, you trade manual flexibility for automated consistency. You're deciding where you want the complexity to live—in your discipline and processes, or in your tools and type systems. Neither is free, but at least with tools, the computer can be wrong at 3 GHz instead of you being wrong at coffee-break speeds.
